# Advanced Guide: `registerMergedGraph`

This document dives deep into the merged-graph workflow exposed by
`@vizij/orchestrator-wasm`. It builds on the main tutorial and explains how to compose multiple
graph specs into a single controller, diagnose errors, and integrate the resulting pipeline into
real-world applications.

---

## 1. When to Merge Graphs

### Use merged graphs to:
- Run producer and consumer graphs in one pass without hitting the blackboard.
- Eliminate frame delays caused by staging outputs and re-reading them.
- Maintain deterministic execution order inside a single controller.
- Hide intermediary values from hosts while still exposing final outputs.

### Avoid merging when:
- You need blackboard persistence across frames.
- Graphs represent independent subsystems (e.g. separate teams own them).
- Multiple graphs intentionally write to the same `TypedPath`.

---

## 2. API Overview

```ts
const orchestrator = await createOrchestrator({ schedule: "SinglePass" });

const mergedId = orchestrator.registerMergedGraph({
  id: "motion-bundle",
  graphs: [
    {
      spec: ioGraphSpec,
      subs: { outputs: ["shared/value"] },
    },
    {
      spec: computeGraphSpec,
      subs: { inputs: ["shared/value"], outputs: ["shared/doubled"] },
    },
  ],
  strategy: {
    outputs: "namespace",
    intermediate: "blend",
  },
});
```

- `id` is optional; autogenerated IDs follow the `graph:<counter>` pattern.
- Each entry in `graphs` mirrors `GraphRegistrationConfig` (`spec`, `subs`).
- `strategy` (optional) lets you decide how to resolve overlapping paths:
  - `"error"` (default) preserves the legacy behaviour and aborts the merge when conflicts occur.
  - `"blend"` injects a `default-blend` node (plus equal weights) so downstream consumers see a single averaged value.
  - `"namespace"` renames final output paths to `graphId/original/path` so parallel values remain distinct.
- The method throws on invalid specs or merge conflicts. Always wrap in `try/catch`.

## 3. Namespacing Mechanics

Internally, the wasm binding calls `GraphControllerConfig::merged`, which assigns namespace prefixes
to every node ID:

```
g0_source::node_name   // graph 0
g1_consumer::node_name // graph 1
```

- Non-alphanumeric characters convert to `_`.
- Conflicts receive a deterministic suffix (`__1`, `__2`, …).
- Namespacing is visible when serializing the merged spec for debugging.

---

## 4. Input/Output Rewiring

During merging, the orchestrator:

1. Finds every `Output` node with a `params.path`.
2. Tracks its upstream edge (source node + output port).
3. Searches for `Input` nodes whose `params.path` matches.
4. Replaces the input node in all outbound edges with the upstream node, removing the original
   `Input` node if it is fully satisfied by the merge.
5. Composes selectors (binding selector first, consumer selector second).

Any inputs still unmatched remain in the merged spec and continue to receive staged values from the
host, so you can mix direct wiring with blackboard-driven inputs.

---

## 5. Inspecting the Merged Spec

You can normalize the merged graph for inspection:

```ts
const merged = orchestrator.registerMergedGraph({ graphs: [...] });
const mergedSpec = await orchestrator.normalizeGraphSpec({ id: merged, spec: {} }); // returns JSON
console.log(JSON.stringify(mergedSpec, null, 2));
```

Alternatively, call the private wasm binding:

```ts
const module = await import("@vizij/orchestrator-wasm/pkg/vizij_orchestrator_wasm.js");
```

and inspect `module.VizijOrchestrator` internals, though this is primarily for debugging.

## 6. Understanding Errors

### Duplicate outputs

```
Error: graph merge error: output path 'shared/value' is produced by multiple graphs: ["graphA","graphB"]
```

Resolution:
- Rename the `params.path` in one of the graphs.
- Keep the conflicting graph as a separate controller if both outputs are required.

### Missing upstream

```
graph merge error: output node 'g0_graphA::publish' in graph 'graphA' is missing an upstream connection
```

Resolution:
- Verify the graph spec includes edges into every `Output` node.
- Check for typos in node IDs within `edges`.

### Empty bundle

```
register_merged_graph: expected at least one graph
```

Self-explanatory; supply at least one graph spec.

Wrap merges in try/catch:

```ts
try {
  orchestrator.registerMergedGraph({ graphs: [...] });
} catch (err) {
  console.error("Merge failed", err);
}
```

## 7. Subscriptions & Mirror Writes

During merge, subscriptions are consolidated:

- Inputs: union of all source `subs.inputs`, minus paths satisfied by merged bindings.
- Outputs: union of all `subs.outputs`.
- Mirror writes: `true` if any source graph sets `mirrorWrites`.

Tips:
- Explicitly declare `subs` on each graph before merging so the result has precise staging.
- After merging, call `orchestrator.listControllers()` and inspect the merged ID’s behavior by
  stepping the orchestrator and examining `frame.merged_writes`.

## 8. Advanced Example

```ts
const mergedId = orchestrator.registerMergedGraph({
  graphs: [
    {
      id: "pose-sampler",
      spec: {
        nodes: [
          { id: "time", type: "input", params: { path: "driver/time.seconds" } },
          { id: "osc", type: "oscillator", params: { frequency: 0.5 } },
          { id: "publish", type: "output", params: { path: "shared/motion" } },
        ],
        edges: [
          { from: { node_id: "time" }, to: { node_id: "osc", input: "phase" } },
          { from: { node_id: "osc" }, to: { node_id: "publish", input: "in" } },
        ],
      },
      subs: {
        inputs: ["driver/time.seconds"],
        outputs: ["shared/motion"],
      },
    },
    {
      id: "rig-controller",
      spec: {
        nodes: [
          { id: "shared", type: "input", params: { path: "shared/motion" } },
          { id: "remap", type: "remap", input_defaults: { in_min: 0, in_max: 1, out_min: -1, out_max: 1 } },
          { id: "publish", type: "output", params: { path: "rig/arm.position" } },
        ],
        edges: [
          { from: { node_id: "shared" }, to: { node_id: "remap", input: "value" } },
          { from: { node_id: "remap" }, to: { node_id: "publish", input: "in" } },
        ],
      },
      subs: {
        inputs: ["shared/motion"],
        outputs: ["rig/arm.position"],
      },
    },
  ],
});

// Stage driver input
orchestrator.setInput("driver/time.seconds", { float: 1.0 });

// Evaluate
const frame = orchestrator.step(1 / 60);
console.log(frame.merged_writes);
```

`shared/motion` never touches the blackboard. Both graphs run in a single pass, and only the final
`rig/arm.position` write is exposed externally.

## 9. Testing Strategies

1. **Local test harness**:

   ```ts
   function readWrite(writes: Array<{ path: string; value: unknown }>, path: string): number {
     return valueAsNumber(writes.find((w) => w.path === path)?.value as ValueJSON)!;
   }

   const frame = orch.step(1 / 60);
   expectNearlyEqual(readWrite(frame.merged_writes, "shared/doubled"), 2);
   ```

2. **Fixtures**:
   Add merged scenarios to Vitest/Vitest tests. The repository’s `npm/@vizij/orchestrator-wasm/tests/all.test.ts`
   showcases a real example (`testMergedGraphRegistration`).

3. **Snapshot the merged spec**:
   Serialize and store the normalized graph for regression tests (helps detect schema drift).

---

## 10. Best Practices

- Keep graphs small and focused, then merge where necessary.
- Provide meaningful `id` values; they appear in namespace prefixes and error messages.
- Normalize specs with `normalizeGraphSpec` before persisting or diffing.
- Catch merge errors early, surface them in tooling, and offer quick fixes (e.g., rename outputs).
- Combine merging with `normalizeGraphSpec` to build pipelines that can be exported/imported across
  Rust and JS environments.

By mastering `registerMergedGraph`, you can craft single-pass pipelines that remain transparent,
assertable, and high-performance across browser and Node runtimes. Happy orchestrating! ♻️🎛️
