# BuildRecord Node â€“ Implementation Spec

## Overview

Add a `BuildRecord` node to the vizij-rs node graph that assembles a `Value::Record(HashMap<String, Value>)` from a variadic set of `Any`-typed inputs, where each input slot carries a user-editable string key.

This document covers all changes required across three layers:
1. **vizij-rs / Rust core** â€“ types, schema, eval, WASM set_param
2. **vizij-rs / TypeScript bindings** â€“ generated types and normalization
3. **vizij-web / Frontend** â€“ UI for per-slot key editing and graph serialization

## Why `Value::Record` and not `arora_schema::KeyValue`

`arora_schema::KeyValue` is a type from a completely separate, unrelated crate (`C:\repo\semio\arora-schema`) that vizij-rs has **zero dependency on** â€” there is not a single mention of "arora" in any `Cargo.toml` in the vizij-rs workspace.

Beyond the dependency issue, `arora_schema::KeyValue` is a heavier, schema-metadata type â€” every container and every field carries a `Uuid` for traceability, and its values are typed by arora's own `Value` enum (covering Rust primitives, structures, enumerations, etc.), not by vizij's animation-centric `vizij_api_core::Value`.

`Value::Record(HashMap<String, Value>)` is the correct, already-established analog within vizij-rs: lightweight, no UUID overhead, and fully consistent with vizij's `Value` type. It is already emitted by the URDF IK nodes today (`UrdfIkPosition`, `UrdfIkPose`).

### 1.5 Add eval logic

**File:** `crates/node-graph/vizij-graph-core/src/eval/eval_node.rs`

**Step A â€“ add the arm in the main dispatch match:**

```rust
NodeType::BuildRecord => eval_build_record(params, inputs, outputs),
```

**Step B â€“ implement `eval_build_record`:**

```rust
fn eval_build_record(
    params: &NodeParams,
    inputs: &InputSlots,
    outputs: &mut OutputSlots,
) -> Result<(), String> {
    let slots = inputs.variadic("field");
    let keys = params.record_keys.as_deref().unwrap_or(&[]);

    let mut map = HashMap::with_capacity(slots.len());
    for (i, pv) in slots.iter().enumerate() {
        let key = keys
            .get(i)
            .filter(|k| !k.is_empty())
            .map(|k| k.as_str())
            .unwrap_or_else(|| "field")
            .to_string();
        // Deduplicate by appending index if the key was already inserted
        let final_key = if map.contains_key(&key) {
            format!("{key}_{i}")
        } else {
            key
        };
        map.insert(final_key, pv.value.clone());
    }

    single_output(outputs, Value::Record(map))
}
```

> `inputs.variadic("field")` follows the same pattern as `collect_operand_ports` (which calls `inputs.variadic("operand")`), returning slots in stable index order matching the serialized `record_keys` array.

### 1.6 Add `"record_keys"` arm to `set_param_inner`

**File:** `crates/node-graph/vizij-graph-wasm/src/lib.rs`

`set_param_inner` has an explicit match arm for each known param key (line ~1324). Without a `"record_keys"` arm, `set_param(nodeId, "record_keys", ...)` hits the catch-all `_ => Err("unknown key")`.

Add the arm directly after the `"case_labels"` arm, mirroring it exactly:

```rust
"record_keys" => {
    node.params.record_keys = Some(parse_string_list(node_id, key, &val)?);
}
```

This is **non-structural**: the `structural_change` flag (line 1241) is only set for `(NodeType::Split, "sizes")`. Changing key strings does not affect port layout or slot count, so the plan cache remains valid â€” exactly the same behaviour as `case_labels`.

## Part 3 â€“ vizij-web Frontend Changes

**File:** `apps/vizij-authoring/src/motiongraph/components/GraphNode.tsx`

### 3.1 Handle label stays as "Field N"; key input is a separate inline element

`buildVariadicPorts` and its `useMemo` deps (`[variadicSpec, variadicCount]`) do **not** change. The `PortSpec.name` (handle label) remains `"Field 1"`, `"Field 2"`, etc.

When `variadicSpec.keyed === true`, render a small text `<input>` element **inline next to each variadic handle** (inside the existing variadic port render loop). This is purely a side element â€” it does not affect `PortSpec.name`, the memo, or ReactFlow handle identity.

Data-flow: on each keystroke the input calls `updateParam("record_keys", newKeysArray)`, which spreads the full new array into `n.data.params.record_keys` via the existing `updateParam` callback. This is consistent with how all other params are updated.

### 3.2 Sync `record_keys` on add/remove

`record_keys` may be `undefined` when a node is first created â€” use `?? []` guards throughout. Do **not** eagerly pre-populate on node creation; the Rust eval handles a missing/null array gracefully (falls back to `"field"` for every slot).

In `addVariadicInput`:
```typescript
const newKeys = [...(data.params?.record_keys ?? []), ""];
updateParam("record_keys", newKeys);
// existing: setNodes(... variadicInputCount + 1 ...)
```

In `removeVariadicInput`:
```typescript
const newKeys = (data.params?.record_keys ?? []).slice(0, -1);
updateParam("record_keys", newKeys);
// existing: setNodes(... variadicInputCount - 1 ...)
```

### 3.3 Graph serialisation

No additional changes. `record_keys` round-trips through `load_graph` / `set_param` automatically because it lives in the standard `params` bag.
